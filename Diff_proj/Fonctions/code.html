Parlons maintenant des méthodes d'instances !

Une méthode d'instance est une fonction associées à une classe et qui s'opère à l'intérieurxde chaque instance de cette classe.

Pour se faire, nous créons une classe : "bankAccount" que nous manipulerons pour rendre celà visuel.

Il faut penser qu'un objet est un tyoe de référence, il est toujour possubel d'apporter des modification à une instance. La partie constante est une référence à cette instance. La méthode ligne 92 utlise le mot clé "this" pour accéder à la "balance".
/!\Dans le corps d'une classe, le mot clé this fait référence à l'instance créée de la classe. Dans cet exemple, il fait référence à "newAccount"./!\

Les méthodes statiques :

La méthode statique n'est pas liée à une instance en particulier de la classe mais à la classe elle-même.
On l'utilise pour créer des méthodes utilitaires ("helper"). Exemple : l'objet "Math". RDV Ligne 232

Ces fonctionnalités pourraient être des fonctions mais elles utilisent des classes statiques que l'on peut regrouper par catégories ou type.

Ainsi, nous avons vu Deux méthodes :

  • les méthodes d'instance, qui agissent sur les instances individuelles d'une classe ;
  • les méthodes statiques, qui ne s'appuient pas sur une instance d'une classe.

Règles pour faire des fonctions propres !

Il est important d'écrire une fonction propre pour trouver facilement les choses que l'on veut dans notre code. "Toujours coder en sachant que la personne qui va nous relire est un psychopathe sachant ou on vit."
Règles d'une fonction propre :

  • Ne pas se répéter dans le code (DRY : Don't Repeat Yourself) ! Une répétition => Une fonction appelable plus facilement. EX ligne 270
  • Bien nommer sa fonction pour décrire ce qu'elle fait ! Comme ça tout est bien littéral. (Ligne 385)
  • Penser à laisser des commentaires. Un bon code est un code commenté pour clarifier quelque chose de non évident. J'ai utilisé l'exemple précédent pour montrer ce qu'est un bon commentaire.
  • Ecrire avec style avec des conventions :
    • les fonctions sont écrites en "camel case" minuscule ("getWordCount" ou "numberOfCats")
    • les noms de classes sont écrites en "camel case" majuscule ("PremiumAccount" ou "SpecialGuest")
    • Le choix de nom est primordiale, il ne doit pas être raccourcis pour faciliter la lecture.
    • les conventions ne sont pas obligatoires mais facilitent la lecture. Ainsi la fonction "comptage de lettre peut devenir tout à fait obscure : ligne 448
  • Mise en forme du code :
    • Lise en retrait à l'aide de l'indentation
    • Espacement entre les lignes et les fonctions pour faire comprendre que pas tout n'est à la suite.
    • Positionnement des accolades ex ligne 459

Tester une Fonction !

L'intérêt de tester les fonction est de s'assurer de l'exactitude du code même ! A l'aide de structures de test descend, nous nous assurons de l'exactitude du code au dur et à mesure.
Il existe ainsi 3 types de test :

  • Création de Test Unitaires : vérifient des unités individuelles (en général des fonctions uniques ou des classes) en leur fournissant une entrée et en s'assurant qu'elles donnent la sortie attendue. Ligne 482 (60-80% du temps)
  • Les tests d'intégrations : Vérification des multiples fonctions ou classes pour s'assurer qu'elles peuvent travailler ensemble. Parfois, l'intégration de deux bonnes fonctions dans un code vont "s'empétrer" l'une dans l'autre, il faut donc s'assurer le la "symbiose"
  • Les tests fonctionnels : Ont pour intérêts de vérifier différents scénarios complexes en contexte. Pour se faire, nous utilisons des applications dédiés.
